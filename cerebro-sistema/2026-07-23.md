# 2026-07-23 — Instruções e decisões

## Regras dadas pelo Kauã (valem sempre)

- **Separar as APIs iFood e 99Food.** Cada loja/plataforma usa a SUA própria API. Se é iFood, API iFood; se é 99, API 99. **Nunca** reaproveitar funcionalidade/rota/formato de uma plataforma dentro do fluxo da outra. (Repreensão: eu estava querendo usar algo da 99 no iFood.)
- **KDS de produção = Flask** (`server/app.py` + `viewer/`). **Ignorar** o `sistema-pedidos/` (FastAPI) — não é o de produção.

## Feito hoje

- **Bug KDS (itens não apareciam):** `get_pedido_detalhe` agora normaliza itens por plataforma na leitura (iFood de `detalhes_brutos.items`); card do KDS usa `itensNormalizados`. Fim do "fica de dois jeitos".
- **Homologação Merchant+Catalog:** auditados os 6 cenários — tudo já implementado ponta a ponta. Hardening: IDs de complemento com fallback, badge "Disponível/Pausado" no complemento, tooltip do item.
- **Catálogo com imagens:** miniatura por item na coluna ITEM (`CatalogoView.jsx`), com placeholder pra item sem foto. `item.foto` já vem como URL exibível do `/api/catalogo`.
- **Listar lojas (híbrido):** mantém lista do Supabase + selo "verificada no iFood" cruzando com `GET /merchants` (novo endpoint `/api/lojas/verificacao-ifood`, não-bloqueante).

## Combo (COMBO_V2) — DESATIVADO por ora

- `POST /api/itens/combo` dá **409 do iFood** "product already has this option group". Causa: formato do payload COMBO_V2 nunca foi confirmado (o próprio código admite) + o grupo principal escolhido estava vazio (um combo precisa que o grupo principal tenha opções/produtos).
- Combo **NÃO está no checklist de homologação** → decisão do Kauã: **ocultar o botão** e consertar depois. Flag `COMBO_HABILITADO = false` em `viewer/src/views/CatalogoView.jsx` (voltar pra `true` quando o create_combo for arrumado).

## Em andamento / próximo

- **Bug no KDS:** pedidos aparecem no quadro mas **sem os itens solicitados**.
  - Pedidos da tela são iFood reais do sandbox (id UUID, cliente "PEDIDO DE TESTE - kaua...").
  - Causa provável: divergência de formato por plataforma no `viewer/` e no `get_pedido_detalhe`.
    - Card do KDS lê `detalhes_brutos.itens` (português) — mas iFood grava `detalhes_brutos` com `items` (inglês, shape cru da Order API).
    - `DetalhesPedidoModal` lê `item.obs`/`complementos[].nome`, mas os itens iFood da tabela `itens_pedido` usam `observacao` e `options` (com `name`).
    - `abrirDetalhes` mostra os itens na hora (do `detalhes_brutos`) e depois **sobrescreve** com `/pedidos/{id}`, que pode voltar `itens: []` para iFood → itens somem.
  - Correção respeitando a separação: normalizar cada shape só na LEITURA (estender o fallback de `get_pedido_detalhe` para iFood também), sem uma plataforma chamar a lógica da outra.
